# Plan Schema

## Overview

Plans are the canonical state records for all work in a booping-managed project. Each plan lives as a single Markdown file with YAML frontmatter under `~/Claude/{project}/plans/*.md`. One file = one plan; the filename is derived by `/groom` from the creation date and a slug (e.g. `20260422-my-feature.md`). State moves through 9 defined lifecycle states, always driven by the `booping-plans` CLI — never by hand-editing frontmatter directly.

The `sprints.md` file at `~/Claude/{project}/sprints.md` is a derived rollup regenerated by `booping-plans sync-sprints --project=<name>` from the frontmatter of every plan file. It is a read-only view, not a source of truth. Any attempt to "fix" `sprints.md` by hand will be overwritten on the next `sync-sprints` run.

## Statuses

States are listed in normal lifecycle order. Terminal states (`done`, `fail`, `cancelled`) appear at the end.

**`backlog`** — The plan file exists but has not yet entered active spec'ing. This is the starting state for every new plan; `/groom` writes plans here initially. A plan may also return to `backlog` from `in-spec` if the user pauses grooming to revisit later, or from `cancelled` if it is revived. In this state the plan may be a raw idea, a research stub, or a partial draft. Nothing is committed to a sprint yet.

**`in-spec`** — `/groom` is actively refining this plan: running spec agents, filling out the frontmatter, computing story-point estimates, writing the DoD. The plan stays here until the quality checklist is clean and the user accepts the estimates. If the user pauses grooming, the plan drops back to `backlog`. No development work should begin on an `in-spec` plan.

**`ready-for-dev`** — The spec is accepted: quality checklist is clean, the user has confirmed estimates, and the `sp` and `planned` fields are set. The plan is queued for a developer to claim. `/develop` picks plans from this state; no other skill should touch a `ready-for-dev` plan until it is claimed.

**`in-progress`** — `/develop` has claimed the plan and implementation is underway. The `started` field is auto-filled by the CLI on this transition. The plan stays `in-progress` until all milestones are complete and the final verify block passes, or until an unrecoverable blocker forces a `fail`.

**`awaiting-retro`** — `/develop` has finished all milestones and the final verification passed. The `completed` field is auto-filled by the CLI. The plan is waiting for a retrospective to be written by `/retro`. No further development changes should be made while the plan is in this state.

**`awaiting-learning`** — `/retro` has written the retrospective document and set the `retro` path and `goal` fields. The plan is now waiting for `/learn` to absorb the lessons into the project's CLAUDE.md and lessons directory. This separation from `awaiting-retro` allows retro writing and lesson extraction to happen at different times.

**`done`** — `/learn` has absorbed all lessons from the retro. This is the terminal success state. The plan is moved to the History section of `sprints.md` on the next `sync-sprints` run. No further automated transitions occur.

**`fail`** — `/develop` hit an unrecoverable blocker or intentionally aborted the plan. The `completed` field is auto-filled by the CLI. This is a terminal technical-execution-failure state, distinct from `cancelled` (which is a product/priority decision). A failed plan stays in History as-is; recovery typically means opening a new plan with a different approach.

**`cancelled`** — The user has explicitly shelved the plan before completion. This is a terminal state representing a product or priority decision rather than a technical failure. The `completed` field is auto-filled by the CLI. A cancelled plan can be revived by running `booping-plans set status=backlog` and flowing through the lifecycle again.

## Transitions

| From | To | Trigger | Who sets it |
|------|----|---------|-------------|
| `backlog` | `in-spec` | /groom begins the spec phase (agents run) | /groom via `booping-plans set status=in-spec` |
| `in-spec` | `ready-for-dev` | quality checklist clean + user accepts estimates | /groom via `booping-plans set status=ready-for-dev` (CLI auto-fills `planned`) |
| `in-spec` | `backlog` | user pauses grooming to revisit later | manual `booping-plans set status=backlog` |
| `ready-for-dev` | `in-progress` | /develop picks up the plan | /develop via `booping-plans set status=in-progress` (CLI auto-fills `started`) |
| `in-progress` | `awaiting-retro` | /develop completes milestones + final verify | /develop via `booping-plans set status=awaiting-retro` (CLI auto-fills `completed`) |
| `in-progress` | `fail` | /develop hits an unrecoverable blocker or aborts | /develop via `booping-plans set status=fail` (CLI auto-fills `completed`) |
| `awaiting-retro` | `awaiting-learning` | /retro publishes the retro doc | /retro via `booping-plans set status=awaiting-learning retro=<path> goal=<enum>` |
| `awaiting-learning` | `done` | /learn absorbs lessons + applies CLAUDE.md touches | /learn via `booping-plans set status=done` |
| (any pre-terminal) | `cancelled` | user shelves the plan (priority change, scope folded into another sprint, etc.) | manual `booping-plans set status=cancelled` (CLI auto-fills `completed`) |

Terminal states are `done`, `fail`, and `cancelled` — no further automatic transitions occur from these states.

The user may manually reopen a terminal plan (e.g. `cancelled → backlog`) by running `booping-plans set status=backlog` on the file; no special guard blocks this. This is rare but supported.

## Frontmatter

Full schema for plan frontmatter. Fields are written by the indicated skill or CLI component; skills never pass date literals for auto-filled fields — they omit the key and let the CLI stamp today's ISO date.

| Field | Required | Default | Writer | Purpose |
|-------|----------|---------|--------|---------|
| `title` | yes | — | /groom | Display name |
| `type` | yes | — | /groom | `feature` \| `bug` \| `refactoring` |
| `status` | yes | `backlog` (initial) | /groom, /develop, /retro, /learn | Lifecycle state (9 values, see [## Statuses](#statuses)) |
| `sp` | yes once `ready-for-dev` | — | /groom | Total story points |
| `source` | yes | `ad-hoc` | /groom | Origin of the plan |
| `created` | yes | today | /groom | Immutable creation stamp |
| `planned` | yes once `ready-for-dev` | today-of-transition | CLI auto-fill | Date the plan was committed to a sprint |
| `started` | optional | today-of-transition | CLI auto-fill | Set on `→ in-progress` |
| `completed` | optional | today-of-transition | CLI auto-fill | Set on `→ awaiting-retro`, `→ fail`, `→ cancelled` |
| `retro` | optional | — | /retro | Path to the retro file |
| `goal` | yes once `awaiting-learning` | — | /retro | `success` \| `partial` \| `fail` |
| `business_goal` | features only | — | /groom | User-visible outcome (one sentence) |

### Field notes

**`source`**: Free-form string. Common values: `ad-hoc`, a URL, or `split-from:plans/<primary-plan-filename>.md` when `/groom` writes a sibling-sprint stub via its Split sub-phase.

**`planned` / `started` / `completed`**: Auto-filled by the CLI on their respective transitions when the key is not already set. Explicit `null` is treated as absent. Skills never pass date literals — they omit the key and let the CLI stamp today's ISO date.

**`goal`**: Set by /retro with value `success | partial | fail`. Reflects the PM's verdict on the business goal — distinct from `status`, which reflects the execution-phase outcome. A plan can `status: done` (execution complete) with `goal: partial` (business outcome only partially achieved).

**`business_goal`**: Features only. One sentence phrased as "the problem the user has today and what changes for them after". Empty or null on bug/refactoring plans.

## CLI and fallback

Skills never hand-edit frontmatter or `sprints.md` directly. They call:

```
booping-plans set --project=<P> <plan-path> <key>=<value> [<key>=<value> ...]
booping-plans sync-sprints --project=<P>
```

See [`../bin/booping-plans`](../bin/booping-plans) for the implementation. Exit codes: 0 ok, 1 generic, 2 file-not-found, 3 malformed-frontmatter (non-fatal), 4 missing-project-arg, 5 invalid-enum-value, 6 malformed-frontmatter (fatal on `set`).

If `set` or `sync-sprints` exits non-zero, the skill falls back to hand-editing the frontmatter and prints stderr verbatim so the CLI bug is visible. See each skill's CLI fallback block for the exact fallback steps.

## Lifecycle exceptions

**Reviving a cancelled plan**: A plan with `status: cancelled` can be revived by running `booping-plans set status=backlog` and flowing through the full lifecycle again from the beginning.

**Recovering from fail**: There is no blessed recovery path for a `fail`ed plan. The typical approach is to open a new plan that tries a different implementation strategy, leaving the failed plan in History as a record. The failed plan's file is not deleted or modified.

**Split stubs**: Plans created by /groom's Split sub-phase start with minimal frontmatter (5 fields: `title`, `type`, `status: backlog`, `source: split-from:plans/<primary>.md`, `created`). They intentionally lack `sp` and `planned` until they themselves reach `ready-for-dev` via a later `/groom` pass.

**Manual migration from the legacy directory**: Vaults that predate this refactor keep their plans in a directory named `backlog` (not `plans`). To migrate, run `git mv <legacy-dir>/* plans/` where `<legacy-dir>` is the old directory name, then `booping-plans set --project=<P> <plan-path> status=<correct-status>` on each file. No migration script is provided — the operation is rare, inspectable, and reversible.
