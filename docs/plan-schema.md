# Plan Schema

## Overview

Plans are the canonical state records for all work in a booping-managed project. Each plan lives as a single Markdown file with YAML frontmatter under `~/Claude/{project}/plans/*.md`; the filename is a date-slug (e.g. `20260422-my-feature.md`). Frontmatter is the source of truth — state moves through 9 lifecycle states via `booping-plans` CLI calls, never hand-edits. `sprints.md` is a derived rollup regenerated by `booping-plans sync-sprints`.

---

## Statuses

| Status | Meaning |
|--------|---------|
| `backlog` | Draft or parked idea; `/groom` writes here initially. |
| `in-spec` | `/groom` is actively refining: running agents, filling frontmatter, computing estimates. |
| `ready-for-dev` | Spec accepted; `sp` and `planned` set; queued for `/develop` to claim. |
| `in-progress` | `/develop` has claimed the plan; `started` auto-filled on transition. |
| `awaiting-retro` | All milestones done; `completed` auto-filled; waiting for `/retro`. |
| `awaiting-learning` | Retro written; `retro` and `goal` set; waiting for `/learn` to absorb lessons. |
| `done` | Terminal success. `/learn` has absorbed all lessons. |
| `fail` | Terminal technical failure. `/develop` hit an unrecoverable blocker; `completed` auto-filled. |
| `cancelled` | Terminal product/priority decision. User shelved the plan; `completed` auto-filled. |

---

## Transitions

Transitions are owned by the skills that trigger them — see `/groom`, `/develop`, `/retro`, `/learn` SKILL.md files for the exact `booping-plans set` calls. Each skill documents its own CLI fallback.

`backlog → in-spec → ready-for-dev → in-progress → awaiting-retro → awaiting-learning → done`

`fail` and `cancelled` are terminal escape hatches reachable from most non-terminal states.

## Frontmatter

| Field | Required | Default | Writer | Purpose |
|-------|----------|---------|--------|---------|
| `title` | yes | — | /groom | Display name |
| `type` | yes | — | /groom | `feature` \| `bug` \| `refactoring` |
| `status` | yes | `backlog` | /groom, /develop, /retro, /learn | Lifecycle state (9 values) |
| `sp` | yes once `ready-for-dev` | — | /groom | Total story points |
| `source` | yes | `ad-hoc` | /groom | Origin of the plan |
| `created` | yes | today | /groom | Immutable creation stamp |
| `planned` | yes once `ready-for-dev` | today-of-transition | CLI auto-fill | Date plan was committed to a sprint |
| `started` | optional | today-of-transition | CLI auto-fill | Set on `→ in-progress` |
| `completed` | optional | today-of-transition | CLI auto-fill | Set on `→ awaiting-retro`, `→ fail`, `→ cancelled` |
| `retro` | optional | — | /retro | Path to the retro file |
| `goal` | yes once `awaiting-learning` | — | /retro | `success` \| `partial` \| `fail` |
| `business_goal` | features only | — | /groom | User-visible outcome (one sentence) |

## sprints.md generation

`sprints.md` is wholesale-regenerated by `booping-plans sync-sprints --project=<name>` from plan frontmatter. The Active table contains non-terminal plans sorted by `planned`/`created`; the History table contains terminal plans sorted by `completed`/`planned`/`created` descending. The output is byte-identical on consecutive runs with no changes. The first line is a plain-text regeneration notice.

---

## CLI and fallback

Skills call `booping-plans set --project=<P> <plan-path> <key>=<value>` and `booping-plans sync-sprints --project=<P>`. See `bin/booping-plans --help` for exit codes and full usage. On non-zero exit, each skill falls back to hand-editing frontmatter per its own SKILL.md.

---

## Lifecycle exceptions

- **Migration from legacy**: Vaults that predate this refactor keep plans in a directory named `backlog` (not `plans`). Migrate with `git mv <legacy-dir>/* plans/` then `booping-plans set --project=<P> <plan-path> status=<correct-status>` on each file.
- **Reviving a cancelled plan**: Set `status` back to `backlog` via `booping-plans set` and flow through the lifecycle again.
